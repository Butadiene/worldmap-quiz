# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Japanese-language PWA quiz for learning world countries' positions and names on a blank map (уЎйтю░тЏ│сѓ»сѓцсѓ║). Pure static site Рђћ vanilla JS, no build step, no package manager, no bundler, no tests. D3 and TopoJSON are pulled from CDN at runtime.

## Running / developing

Service Workers do not run over `file://`, so serve over HTTP:

```bash
python3 -m http.server 8000   # then open http://localhost:8000
```

There is no build, lint, or test tooling. Edit the files and reload the browser.

**When changing any local asset, bump the cache version** `CACHE = "worldquiz-vN"` in `sw.js` **and the matching `APP_VERSION = "vN"` in `app.js`** (shown on the setup screen so users can confirm an update landed) Рђћ otherwise returning users keep the old cached files (the SW serves cache-first). Also add any new local file to the `LOCAL_ASSETS` array in `sw.js`, and any new CDN URL to both `CDN_ASSETS` (sw.js) and the `<script>` tags / fetch URL in the app.

## Architecture

Everything is driven by `app.js`, one IIFE holding a single `state` object. There is no framework and no routing Рђћ the UI is a stack of full-screen `<section>`/overlay elements in `index.html` that are shown/hidden by toggling their `hidden` attribute (see `showSetup`, `startQuiz`, `nextQuestion`, `endQuiz`).

**Data joins on ISO 3166-1 numeric country codes**, zero-padded to 3 digits (`pad3`). This is the linchpin connecting three sources:

- `countries-50m.json` (world-atlas, CDN) Рєњ GeoJSON features whose `f.id` is the numeric code. These are the map polygons. (The higher-detail 50m data is used so small island nations have polygons; drop to `countries-110m.json` for lighter borders.)
- `countries.js` Рєњ `window.COUNTRY_DATA[code]` = `{ ja, region }`. **Only countries present here are rendered and quizzed** Рђћ `buildFeatures` filters out any feature lacking a `COUNTRY_DATA` entry. So adding a country to the quiz = adding its padded numeric code to `countries.js`. **Exception:** features with no ISO id (the 50m atlas has 5 Рђћ Somaliland/Kosovo/N. Cyprus/Indian Ocean Ter./Siachen Glacier) are collected into `state.inertFeatures` and drawn as **inert land** (see Map rendering) so continents have no ocean-colored holes; they are excluded from every id-keyed cache (pool, paths, hit-testing, centroids, ADJ) and are completely non-interactive. Id-bearing features lacking a `COUNTRY_DATA` entry are still skipped entirely.
- `explanations.js` Рєњ `window.COUNTRY_INFO[code]` = `{ cap, note }`, used only by the explanation panel (УДБУфгсЃбсЃ╝сЃЅ) after each answer. Optional; missing entries degrade gracefully.

Region keys (`asia`, `europe`, `africa`, `north_america`, `south_america`, `oceania`, `other`) are defined in `countries.js` and labeled via `REGION_LABEL` in `app.js`; they drive both the region chips on the setup screen and pool filtering.

**Quiz flow:** setup screen collects `state.settings` (mode / region / count / explain) Рєњ `buildPool` filters features by region Рєњ `startQuiz` shuffles and slices to `count` Рєњ `nextQuestion` dispatches to one of two modes:

- **find** (`тю░тЏ│сЂДТјбсЂЎ`): show a name, user taps the country polygon (`onCountryClick`).
- **name** (`тљЇтЅЇсѓњжЂИсЂХ`): highlight a country, user picks from 4 choices Рђћ the correct one plus 3 distractors drawn from the same pool (`askName`).
- **recall** (`тљЇтЅЇсѓњТђЮсЂётЄ║сЂЎ`): forced-recall variant of name. Guesses are pure sonar Рђћ they never call `recordAnswer`/`scoreWrong`. Both converge on `finishTurn`, which either shows the explanation panel (when explain is on) or auto-advances after a delay.

**Auto-focus rollback (find/explore):** an answer's zoom-in (wrong-answer "burn-in" `focusFeature`, or the explanation panel's focus) is the *app's* focus, not the player's. Just before that first focus, `savePreFocus()` records the current view as a **geographic anchor** `state.preFocus = { geo:[lon,lat], eff }` Рђћ `geo` is `projection.invert` of the screen center (input is the *pre-transform* projection-plane point `[(cssW/2Рѕњt.x)/t.k, (cssH/2Рѕњt.y)/t.k]`) and `eff = projection.scale()┬иt.k` is the effective zoom (only the first focus of a turn is recorded; an unprojectable center Рђћ null/NaN invert Рђћ is simply not saved, so no rollback). Storing lon/lat instead of a raw transform is deliberate: the explanation panel opening/closing resizes `map-stage` Рєњ `syncSizeNow` Рєњ `buildPaths` (a reprojection), which would invalidate a saved transform every single turn Рђћ the bug this fixes. A geographic anchor survives reprojection. A user gesture (`zoom "start"` with `sourceEvent`) clears `preFocus` (and downgrades `viewMode` `"anchor"Рєњ"fit"`) so their own pan/zoom is never overridden. `nextQuestion` then, for find/explore, rolls `preFocus` back with a 400 ms `zoom.transform` transition to `anchorTransform(preFocus)` (which re-derives the transform under the *current* projection: `k = clamp(eff/scale, 1, 100)`, translate so `projection(geo)` lands at screen center) and sets `viewMode = "anchor"`, `viewAnchor = preFocus`; if `preFocus` is null it leaves the view alone (this replaced Run 2's always-re-fit, the "zoom keeps snapping back" annoyance), and if the anchor fails to project it falls back to a fresh `fitToFeatures`. **`reapplyView()` handles `"anchor"`** by re-applying `anchorTransform(viewAnchor)` non-animated Рђћ so a later resize (the panel close) re-derives the *same* view under the new projection instead of losing it. `fitToFeatures` resets `viewMode="fit"` and nulls both `preFocus` and `viewAnchor`. NAME mode is untouched (it `focusFeature`s every question by design, `viewMode="focus"`); journey has its own engine.

**Map rendering** uses D3 `geoNaturalEarth1` projected onto an HTML `<canvas>` (`#map`) Рђћ there are no per-country DOM nodes. **The projection only changes when the map is re-framed** (fit/resize), so pan/zoom is pure `ctx`-transform work, never reprojection. `buildPaths()` reprojects every country once per fit into a **Path2D cache**: `paths` (id Рєњ Path2D), `landPath` (all countries in one Path2D for the batched fill/stroke), and `boundsMap` (id Рєњ projected bbox, for culling). It uses a **separate** `pathGen = d3.geoPath(projection)` so the `ctx`-bound `geoPath` (kept only for `centroid`/`bounds`/`area`) is never repointed at a Path2D. `buildPaths()` must be called after every `projection.fitExtent` Рђћ i.e. in `fitToFeatures` and `syncSizeNow`, the only two places the projection changes.

`buildPaths()` also builds `inertPath` (all `state.inertFeatures` in one Path2D) the same way as `landPath`. `render()` draws `inertPath` in plain land color as **step 0** (below everything), and `blit()`'s exposed-ring redraw does the same before its per-country loop, so inert land never leaves a hole. Because inert features are absent from `paths`, `featureAt` naturally treats a tap on them as ocean.

**LOD (2-level detail)**: `loadWorld()` also fetches the coarse `countries-110m.json` in parallel via `loadLowDetail()` Рђћ best-effort, its own promise chain, so a slow/failed 110m never blocks or breaks the app. `ingestLowTopo()` extracts it once into `lowById` (merged id Рєњ 110m feature, same Australia-dedup/`id==null` rules as `buildFeatures`) + `lowInert`, touching **no app state** (LOD is a pure draw-layer alternate). `buildPaths()` then also builds a low-detail set Рђћ `pathsLow`, `landPathLow`, `inertPathLow` Рђћ under the **same projection/`pathGen`**; ids absent from 110m (e.g. the 12 island nations only in 50m) reuse their 50m Path2D. The selector `useLow()` returns true when 110m is loaded **and** `effScale = projection.scale() * t.k < DETAIL_SCALE` (180 Рђћ world fit РЅѕ 70, region fits РЅѕ 300, so only the world view and its zoom-in ramp use low detail). `activePaths()`/`activeLand()`/`activeInert()` wrap it; **`render()`, `blit()`'s exposed draw, and `featureAt()` all read the same selector**, so the drawn pixels and the hit-test never disagree on detail level (the pixel-exact-tap invariant). When 110m arrives after the 50m map is up, `loadLowDetail()` re-runs `buildPaths()` + `render()` to enable LOD live. Crossing the threshold shifts borders ~1px (normal LOD behavior); `boundsMap` stays 50m-only (culling tolerates the sub-pixel bbox difference).

`render()` clears, applies `dpr`+zoom to the context, then: (1) fills/strokes the land Рђћ one `ctx.fill(activeLand())`+`ctx.stroke(...)` near the world view, or per-country culled draws when `t.k > 1.2` (only countries whose `boundsMap` bbox meets the viewport, via `onScreen`); in ТѕљуИЙсЃъсЃЃсЃЌ mode (`state.paint`) it fills each country (from `activePaths()`) with its bucket color and strokes `activeLand()` once. (`activeLand`/`activePaths`/`activeInert` pick the 50m or 110m set per the LOD selector above; near the world view that's the 110m batch.) (2) Overpaints the few "marked" countries (`state.marks`: id Рєњ `target`/`correct`/`wrong`/`hi`) opaquely on top. **Marked countries stay inside `landPath`** Рђћ the overpaint is opaque, so `landPath` never depends on `state.marks` and needs no per-answer rebuild. (3) Draws browse labels unscaled in screen space. **The per-country fill-color decision (mark > ТѕљуИЙсЃъсЃЃсЃЌ bucket > land) lives in one place, `countryFill(id)`, and the per-country fill+stroke in `drawCountry(id, strokeW)`** Рђћ used by both the culled `render()` passes and `blit()`'s exposed-ring draw so the two can't diverge. (The batched `landPath` fill/stroke near the world view is the only non-`drawCountry` land path, and it's plain land color with marks overpainted on top.) `d3.zoom` stores the transform in `state.transform`; the name-mode pulse is a `requestAnimationFrame` loop (`startPulse`/`stopPulse`, cheap because culling kicks in when zoomed). Colors come from the CSS `:root` tokens via `readColors()`. Backing-store `dpr` is capped at 2 (`deviceRatio()`) so 3x phones don't render 2.25├Ќ the pixels for no visible gain. `fitToFeatures` reframes the projection to a set of features; `focusFeature` zoom-pans to a single country (`applyFocus` caps the focus scale at 60, deep enough to read tiny island nations). Zoom `scaleExtent` is `[1, 100]` Рђћ culling and the blit re-anchor absorb the draw cost at depth. **`focusFeature` re-anchors before it glides**: because an answer's focus often follows a `map-stage` resize (explanation panel / choices toggling), it snapshots the current view via `currentViewAnchor()` (the reprojection-proof `{geo, eff}` also used by `savePreFocus`), and if `syncSizeNow()` rebuilt the projection it instantly reconstructs that same geographic view with `anchorTransform` *before* starting the focus transition Рђћ otherwise the first frame would be "old transform ├Ќ new projection" (a different place) and the glide would visibly jump from there.

**Gesture / transition blitting (`blit`)**: re-rendering all polygons every frame of a pan/pinch was the mobile bottleneck, so while the map is moving the last sharp frame is *transformed*, not redrawn Рђћ and this now covers **programmatic zoom transitions** too, not just finger gestures. `captureSnap()` copies the live canvas into an offscreen buffer plus the transform it was taken at (`snap.t0`) **and the projection generation `pathsGen`** (`buildPaths()` bumps `pathsGen` every fit). It runs on zoom `"start"` (finger gesture) and via `beginTransition()` Рђћ which forces one sharp `render()` then snapshots Рђћ just before every programmatic glide: `applyFocus` (animated), the zoom buttons, and the auto-focus rollback. The zoom handler is now simply: **blit if a same-generation snapshot exists (`snap.gen === pathsGen`), else full `render()`**. So finger pans/pinches, inertial fling steps, and transition ticks all glide by blitting, while a *fit* (which rebuilt the projection, bumping `pathsGen`) has a stale snapshot and full-renders every frame Рђћ the correct behavior. `blit()` maps the snapshot to the live view with `r = t.k/t0.k`, `offset = t Рѕњ r┬иt0` (derivation in the code). Because the snapshot only holds what was on screen when it was taken, panning/pinch-out reveals areas that were captured off-screen; leaving them blank was a visible ~2 s lag, so `blit()` also **live-draws the exposed region every frame**. It computes the snapshot's on-screen coverage rect R: if R fully covers the viewport (px guard) it just transfers; if coverage drops below 50% **or the snapshot has been scaled up past 1.5├Ќ (`r > 1.5`, i.e. a zoom-in transition / pinch-in going soft)** it re-anchors with a sharp `render()` + fresh `captureSnap()` for that one frame; otherwise it transfers, then draws `viewport Рѕњ R` (an `evenodd` `Path2D` clip of the viewport РіЋ R) using the same `drawCountry` code `render()` uses Рђћ so revealed land, mastery paint, and answer marks all match exactly. The per-country **selection** tests each bbox against the **exposed bands** Рђћ the up-to-4 rectangles of `viewport Рѕњ (RРѕЕviewport)`, via `bboxIntersects` (the rect-taking generalization of `onScreen`) Рђћ not the whole viewport, so a normal thin-strip pan streams only a handful of countries. The move always terminates on a sharp `render()`: a gesture's `startFling()` renders when it declines to glide and the fling loop renders once the glide settles; a programmatic transition renders at zoom `"end"` (`else render()`). The name-mode **pulse pauses while a transition runs** (`transitioning` flag, set in `beginTransition`, cleared at every zoom `"end"`) so its per-frame `render()` doesn't fight the blit; it resumes when the transition ends. `snap` is nulled whenever the backing store resizes (`resizeCanvas`/`syncSizeNow`). Accepted artifacts (cleared by the next `render`): pinch/zoom-in scaling is momentarily soft (bounded by the `r > 1.5` re-anchor), and browse labels / journey route lines aren't drawn in the exposed ring (they stretch with the map inside R and reappear on the next `render`).

**Click hit-testing** (`featureAt`, used by find mode and browse) has no DOM targets. It is pixel-exact: it tests the pointer against the **same cached Path2D** objects `render()` draws (via the same `activePaths()` LOD selector), under the *same* `dpr`+zoom transform, with `ctx.isPointInPath(activePaths().get(id), mx*dpr, my*dpr)` (backing-store px). A near-miss fallback strokes each path with a fat pen (`lineWidth = 16/t.k`, a constant ~8 CSS px) and uses `ctx.isPointInStroke`, picking the smallest-area country so tiny islands beat big neighbours. This deliberately avoids `projection.invert` + `d3.geoContains`, whose iterative inverse and great-circle edges disagree with the rendered straight-segment borders by a pixel or two (that mismatch was a real off-by-a-bit tap bug).

**Regions** come in two layers: the 7 coarse continents live in `countries.js` (`region` field), while finer sub-regions (ТЮ▒тЇЌсѓбсѓИсѓб, ТЮ▒сЃесЃ╝сЃГсЃЃсЃЉ, тїЌсѓбсЃЋсЃфсѓФ, сѓФсЃфсЃќ, Рђд) are defined in `app.js` as `SUBREGIONS` Рђћ keyed by sets of ISO numeric codes so `countries.js` stays untouched. `inRegion(f, r)` resolves either layer; the setup screen groups the chips by continent.

**Browse/study mode** (`state.browsing`, mode `browse`) is a non-quiz view: `startBrowse` shows the region's map with country-name labels drawn on the canvas (`buildLabels` precomputes centroids/widths per fit; `render()` draws them unscaled in screen space, hiding labels for countries too small at the current zoom). Tapping a country (`showBrowseDetail`) reuses the explanation panel to show capital + blurb.

**Offline:** `sw.js` caches the app shell + CDN libs on install (local assets must all succeed; CDN cached best-effort), then serves cache-first with a navigation fallback to `index.html`.
