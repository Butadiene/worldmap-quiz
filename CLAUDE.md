# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Japanese-language PWA quiz for learning world countries' positions and names on a blank map (白地図クイズ). Pure static site — vanilla JS, no build step, no package manager, no bundler, no tests. D3 and TopoJSON are pulled from CDN at runtime.

## Running / developing

Service Workers do not run over `file://`, so serve over HTTP:

```bash
python3 -m http.server 8000   # then open http://localhost:8000
```

There is no build, lint, or test tooling. Edit the files and reload the browser.

**When changing any local asset, bump the cache version** `CACHE = "worldquiz-vN"` in `sw.js` **and the matching `APP_VERSION = "vN"` in `app.js`** (shown on the setup screen so users can confirm an update landed) — otherwise returning users keep the old cached files (the SW serves cache-first). Also add any new local file to the `LOCAL_ASSETS` array in `sw.js`, and any new CDN URL to both `CDN_ASSETS` (sw.js) and the `<script>` tags / fetch URL in the app.

## Architecture

Everything is driven by `app.js`, one IIFE holding a single `state` object. There is no framework and no routing — the UI is a stack of full-screen `<section>`/overlay elements in `index.html` that are shown/hidden by toggling their `hidden` attribute (see `showSetup`, `startQuiz`, `nextQuestion`, `endQuiz`).

**Data joins on ISO 3166-1 numeric country codes**, zero-padded to 3 digits (`pad3`). This is the linchpin connecting three sources:

- `countries-50m-jpn.json` (repo-hosted) → GeoJSON features whose `f.id` is the numeric code. These are the map polygons. (The higher-detail 50m data is used so small island nations have polygons.) **Map POV (v42, official-data-driven since v43):** the app ships a "Japan point-of-view" transform of world-atlas — `tools/make-jpn-pov.cjs` compares Natural Earth's official default vs Japan-POV 10m files (`ne_10m_admin_0_countries[_jpn]` — POV variants exist only at 10m, which is why world-atlas can't be swapped for an official POV build) and, for each world-atlas polygon whose ownership differs between the two (centroid point-in-polygon, with outline-nearest + current-owner-disappearance fallbacks for islets the 10m data omits, e.g. the westernmost Habomai islets), reassigns its TopoJSON arc references to the Japan-POV owner, dissolving shared borders on the receiving side via `topojson.mergeArcs` (topology/coords untouched). Result at 50m: Northern Territories Russia(643)→Japan(392), Crimea Russia→Ukraine(804), and the formerly-inert Somaliland→Somalia(706), N. Cyprus→Cyprus(196), Siachen Glacier→India(356); at 110m only Crimea/Somaliland/N. Cyprus (no NT/Siachen polygons there). Takeshima/Senkaku are Japan in the official POV but have no polygons at 50m/110m, so they can't be drawn. The full verified de-facto vs Japan-POV diff — including the two sub-polygon cases the transform cannot fix (western Western Sahara stays Morocco, Golan stays Israel) — is recorded in `tools/japan-pov-differences.md`. The selector is `MAP_POV` in `app.js` **and a matching const in `sw.js`** — `"jpn"` loads the two repo-hosted `-jpn` files, `"default"` loads the original world-atlas CDN files (de-facto worldview). **To revert: flip both MAP_POV consts to "default", bump CACHE/APP_VERSION, and restore the setup-credit wording** (the current credit text mentions the Japan POV). The `.setup-credit` text must always match the active POV. **The owner explicitly wants user-facing text to say no more than "Natural Earth の日本視点データに合わせて調整" — no per-territory enumeration (北方領土/クリミア/竹島 etc.) in the app UI; that detail lives only in internal files (README, THIRD_PARTY_NOTICES.md, code comments, this file).**
- `countries.js` → `window.COUNTRY_DATA[code]` = `{ ja, region }`. **Only countries present here are rendered and quizzed** — `buildFeatures` filters out any feature lacking a `COUNTRY_DATA` entry. So adding a country to the quiz = adding its padded numeric code to `countries.js`. **Microstates (v31):** the 18 smallest countries (Vatican, Monaco, …, Tuvalu — the `MICRO_POS` table in `app.js`) are in `COUNTRY_DATA` and quizzed like any other, but are *displayed* as constant-size circle markers while their real polygon is too small on screen (see Map rendering). Tuvalu (798) has no polygon in the 50m atlas at all, so `buildFeatures` synthesizes a tiny diamond feature around its `MICRO_POS` anchor (winding order matters: W→N→E→S keeps the inside small on the sphere) — it then flows through every id-keyed cache like a normal country. **Exception:** features with no ISO id (the jpn 50m file has 2 — Kosovo/Indian Ocean Ter.; the jpn 110m file has only Kosovo; the original world-atlas 50m has 5, but v43 merged Somaliland/N. Cyprus/Siachen Glacier into their Japan-POV owners) are collected into `state.inertFeatures` and drawn as **inert land** (see Map rendering) so continents have no ocean-colored holes; they are excluded from every id-keyed cache (pool, paths, hit-testing, centroids, ADJ) and are completely non-interactive. Id-bearing features lacking a `COUNTRY_DATA` entry are still skipped entirely.
- `explanations.js` → `window.COUNTRY_INFO[code]` = `{ cap, note }`, used only by the explanation panel (解説モード) after each answer. Optional; missing entries degrade gracefully.

Region keys (`asia`, `europe`, `africa`, `north_america`, `south_america`, `oceania`, `other`) are defined in `countries.js` and labeled via `REGION_LABEL` in `app.js`; they drive both the region chips on the setup screen and pool filtering.

**Quiz flow:** setup screen collects `state.settings` (mode / region / count / explain) → `buildPool` filters features by region → `startQuiz` shuffles and slices to `count` → `nextQuestion` dispatches to one of two modes:

- **find** (`地図で探す`): show a name, user taps the country polygon (`onCountryClick`).
- **name** (`名前を選ぶ`): highlight a country, user picks from 4 choices — the correct one plus 3 distractors drawn from the same pool (`askName`).
- **recall** (`名前を思い出す`): forced-recall variant of name. Phase 1 (`askRecall`): highlight the country, prompt「この国・地域の名前は？はっきり思い浮かべてみよう」with a single 思い浮かべた！ button (no names shown). (Question kickers across find/name/recall/explore say 国・地域 (v41) because the dataset includes non-sovereign territories — Taiwan, Palestine, Greenland etc.; general explanatory text and the tutorial keep plain 国 for readability.) Phase 2 (`showRecallChoices`): 6 choices — 5 country names (the correct one plus 4 distractors, always present) and「わからなかった」. わからなかった is an always-wrong honest bailout: counted as a miss (`scoreWrong`/stats) but without the red wrong-flash on the button (`dataset.idk`). Scoring/stats flow through `finishTurn` like name mode. (The old ここにはなかった/80%-present mechanic was removed in v29.)
- **explore** (`たんけん`): **hidden from the setup menu since v21** (button removed from `index.html`, mode dropped from `applySavedSettings`' whitelist; engine code kept). Globle-style. The target is hidden; each tapped country is tinted a heat color by its great-circle distance to the target (`d3.geoCentroid` + `d3.geoDistance`, `state.paint` reused as the heat map, fresh per question in `askExplore`). Guesses are pure sonar — they never call `recordAnswer`/`scoreWrong`. Only discovery (`finishTurn(c, guesses<=4, …)`) or ギブアップ (`scoreWrong` + `finishTurn(c, false, …)`) records stats, exactly once each via `finishTurn`. Survival (count=-1) is treated as a 10-question run here, not lives.

- **journey** (`世界一周`): **hidden from the setup menu since v21** (same treatment as explore; engine code kept). Its own engine (`startJourney`/`newLeg`, not `startQuiz`/`nextQuestion`). Travel from a start country to a goal by tapping **only adjacent countries**; a session is 3 legs where each goal-or-give-up point becomes the next start. Adjacency is `ADJ` (a `Map(id → Set<id>)` built once in `buildAdjacency` from `topojson.neighbors` land borders **plus** the 70 hand-curated `SEA_LINKS` ferry/strait routes). `bfsFrom` picks a 4–9-hop goal and the par move count; visited countries are painted (`--journey-trail`), the route is a polyline over cached projected centroids (`projCentroids`, drawn in `render()` after marks / before labels), the goal is never highlighted (finding it is the game). Stars (3/2/1) score each leg. This mode **never touches `stats`/`mistakes`/`recordAnswer`**; the result "もう一度" (`resultAgain`) branches on `mode` to re-run `startJourney`.

Both converge on `finishTurn`, which either shows the explanation panel (when explain is on) or auto-advances after a delay.

**Auto-focus rollback (find/explore):** an answer's zoom-in (wrong-answer "burn-in" `focusFeature`, or the explanation panel's focus) is the *app's* focus, not the player's. Just before that first focus, `savePreFocus()` records the current view as a **geographic anchor** `state.preFocus = { geo:[lon,lat], eff }` — `geo` is `projection.invert` of the screen center (input is the *pre-transform* projection-plane point `[(cssW/2−t.x)/t.k, (cssH/2−t.y)/t.k]`) and `eff = projection.scale()·t.k` is the effective zoom (only the first focus of a turn is recorded; an unprojectable center — null/NaN invert — is simply not saved, so no rollback). Storing lon/lat instead of a raw transform is deliberate: the explanation panel opening/closing resizes `map-stage` → `syncSizeNow` → `buildPaths` (a reprojection), which would invalidate a saved transform every single turn — the bug this fixes. A geographic anchor survives reprojection. A user gesture (`zoom "start"` with `sourceEvent`) clears `preFocus` (and downgrades `viewMode` `"anchor"→"fit"`) so their own pan/zoom is never overridden. `nextQuestion` then, for find/explore, rolls `preFocus` back with a 400 ms `zoom.transform` transition to `anchorTransform(preFocus)` (which re-derives the transform under the *current* projection: `k = clamp(eff/scale, 1, 100)`, translate so `projection(geo)` lands at screen center) and sets `viewMode = "anchor"`, `viewAnchor = preFocus`; if `preFocus` is null it leaves the view alone (this replaced Run 2's always-re-fit, the "zoom keeps snapping back" annoyance), and if the anchor fails to project it falls back to a fresh `fitToFeatures`. **`reapplyView()` handles `"anchor"`** by re-applying `anchorTransform(viewAnchor)` non-animated — so a later resize (the panel close) re-derives the *same* view under the new projection instead of losing it. `fitToFeatures` resets `viewMode="fit"` and nulls both `preFocus` and `viewAnchor`. NAME mode is untouched (it `focusFeature`s every question by design, `viewMode="focus"`); journey has its own engine.

**Map rendering** uses D3 `geoNaturalEarth1` projected onto an HTML `<canvas>` (`#map`) — there are no per-country DOM nodes. **The projection only changes when the map is re-framed** (fit/resize), so pan/zoom is pure `ctx`-transform work, never reprojection. `buildPaths()` reprojects every country once per fit into a **Path2D cache**: `paths` (id → Path2D), `landPath` (all countries in one Path2D for the batched fill/stroke), and `boundsMap` (id → projected bbox, for culling). It uses a **separate** `pathGen = d3.geoPath(projection)` so the `ctx`-bound `geoPath` (kept only for `centroid`/`bounds`/`area`) is never repointed at a Path2D. `buildPaths()` must be called after every `projection.fitExtent` — i.e. in `fitToFeatures` and `syncSizeNow`, the only two places the projection changes.

`buildPaths()` also builds `inertPath` (all `state.inertFeatures` in one Path2D) the same way as `landPath`. `render()` draws `inertPath` in plain land color as **step 0** (below everything), and `blit()`'s exposed-ring redraw does the same before its per-country loop, so inert land never leaves a hole. Because inert features are absent from `paths`, `featureAt` naturally treats a tap on them as ocean.

**LOD (2-level detail)**: `loadWorld()` also fetches the coarse `countries-110m.json` in parallel via `loadLowDetail()` — best-effort, its own promise chain, so a slow/failed 110m never blocks or breaks the app. `ingestLowTopo()` extracts it once into `lowById` (merged id → 110m feature, same Australia-dedup/`id==null` rules as `buildFeatures`) + `lowInert`, touching **no app state** (LOD is a pure draw-layer alternate). `buildPaths()` then also builds a low-detail set — `pathsLow`, `landPathLow`, `inertPathLow` — under the **same projection/`pathGen`**; ids absent from 110m (e.g. the 12 island nations only in 50m) reuse their 50m Path2D. The selector `useLow()` returns true when 110m is loaded **and** `effScale = projection.scale() * t.k < DETAIL_SCALE` (180 — world fit ≈ 70, region fits ≈ 300, so only the world view and its zoom-in ramp use low detail). `activePaths()`/`activeLand()`/`activeInert()` wrap it; **`render()`, `blit()`'s exposed draw, and `featureAt()` all read the same selector**, so the drawn pixels and the hit-test never disagree on detail level (the pixel-exact-tap invariant). When 110m arrives after the 50m map is up, `loadLowDetail()` re-runs `buildPaths()` + `render()` to enable LOD live. Crossing the threshold shifts borders ~1px (normal LOD behavior); `boundsMap` stays 50m-only (culling tolerates the sub-pixel bbox difference).

`render()` clears, applies `dpr`+zoom to the context, then: (1) fills/strokes the land — one `ctx.fill(activeLand())`+`ctx.stroke(...)` near the world view, or per-country culled draws when `t.k > 1.2` (only countries whose `boundsMap` bbox meets the viewport, via `onScreen`); in 成績マップ mode (`state.paint`) it fills each country (from `activePaths()`) with its bucket color and strokes `activeLand()` once. (`activeLand`/`activePaths`/`activeInert` pick the 50m or 110m set per the LOD selector above; near the world view that's the 110m batch.) (2) Overpaints the few "marked" countries (`state.marks`: id → `target`/`correct`/`wrong`/`hi`) opaquely on top. **Marked countries stay inside `landPath`** — the overpaint is opaque, so `landPath` never depends on `state.marks` and needs no per-answer rebuild. (3) Draws browse labels unscaled in screen space. **The per-country fill-color decision (mark > 成績マップ bucket > land) lives in one place, `countryFill(id)`, and the per-country fill+stroke in `drawCountry(id, strokeW)`** — used by both the culled `render()` passes and `blit()`'s exposed-ring draw so the two can't diverge. (The batched `landPath` fill/stroke near the world view is the only non-`drawCountry` land path, and it's plain land color with marks overpainted on top.) `d3.zoom` stores the transform in `state.transform`; the name-mode pulse is a `requestAnimationFrame` loop (`startPulse`/`stopPulse`, cheap because culling kicks in when zoomed). Colors come from the CSS `:root` tokens via `readColors()`. Backing-store `dpr` is capped at 2 (`deviceRatio()`) so 3x phones don't render 2.25× the pixels for no visible gain. `fitToFeatures` reframes the projection to a set of features; `focusFeature` zoom-pans to a single country (`applyFocus` caps the focus scale at 60, deep enough to read tiny island nations). Zoom `scaleExtent` is `[1, 100]` — culling and the blit re-anchor absorb the draw cost at depth. **`focusFeature` re-anchors before it glides**: because an answer's focus often follows a `map-stage` resize (explanation panel / choices toggling), it snapshots the current view via `currentViewAnchor()` (the reprojection-proof `{geo, eff}` also used by `savePreFocus`), and if `syncSizeNow()` rebuilt the projection it instantly reconstructs that same geographic view with `anchorTransform` *before* starting the focus transition — otherwise the first frame would be "old transform × new projection" (a different place) and the glide would visibly jump from there.

**Gesture / transition blitting (`blit`)**: re-rendering all polygons every frame of a pan/pinch was the mobile bottleneck, so while the map is moving the last sharp frame is *transformed*, not redrawn — and this now covers **programmatic zoom transitions** too, not just finger gestures. `captureSnap()` copies the live canvas into an offscreen buffer plus the transform it was taken at (`snap.t0`) **and the projection generation `pathsGen`** (`buildPaths()` bumps `pathsGen` every fit). It runs on zoom `"start"` (finger gesture) and via `beginTransition()` — which forces one sharp `render()` then snapshots — just before every programmatic glide: `applyFocus` (animated), the zoom buttons, and the auto-focus rollback. The zoom handler is now simply: **blit if a same-generation snapshot exists (`snap.gen === pathsGen`), else full `render()`**. So finger pans/pinches, inertial fling steps, and transition ticks all glide by blitting, while a *fit* (which rebuilt the projection, bumping `pathsGen`) has a stale snapshot and full-renders every frame — the correct behavior. `blit()` maps the snapshot to the live view with `r = t.k/t0.k`, `offset = t − r·t0` (derivation in the code). Because the snapshot only holds what was on screen when it was taken, panning/pinch-out reveals areas that were captured off-screen; leaving them blank was a visible ~2 s lag, so `blit()` also **live-draws the exposed region every frame**. It computes the snapshot's on-screen coverage rect R: if R fully covers the viewport (px guard) it just transfers; if coverage drops below 50% **or the snapshot has been scaled up past 1.5× (`r > 1.5`, i.e. a zoom-in transition / pinch-in going soft)** it re-anchors with a sharp `render()` + fresh `captureSnap()` for that one frame; otherwise it transfers, then draws `viewport − R` (an `evenodd` `Path2D` clip of the viewport ⊕ R) using the same `drawCountry` code `render()` uses — so revealed land, mastery paint, and answer marks all match exactly. The per-country **selection** tests each bbox against the **exposed bands** — the up-to-4 rectangles of `viewport − (R∩viewport)`, via `bboxIntersects` (the rect-taking generalization of `onScreen`) — not the whole viewport, so a normal thin-strip pan streams only a handful of countries. The move always terminates on a sharp `render()`: a gesture's `startFling()` renders when it declines to glide and the fling loop renders once the glide settles; a programmatic transition renders at zoom `"end"` (`else render()`). The name-mode **pulse pauses while a transition runs** (`transitioning` flag, set in `beginTransition`, cleared at every zoom `"end"`) so its per-frame `render()` doesn't fight the blit; it resumes when the transition ends. `snap` is nulled whenever the backing store resizes (`resizeCanvas`/`syncSizeNow`). Accepted artifacts (cleared by the next `render`): pinch/zoom-in scaling is momentarily soft (bounded by the `r > 1.5` re-anchor), and browse labels / journey route lines aren't drawn in the exposed ring (they stretch with the map inside R and reappear on the next `render`).

**Microstate markers (v31)**: the 18 `MICRO_POS` countries render as a circle (white halo + `countryFill(id)` + border) drawn in screen space after the marks / journey line and before labels, anchored at `microXY` (the projected `MICRO_POS` anchor, rebuilt by `buildPaths`). The radius is **capped by Luxembourg's on-screen size** (`microR`): the radius of a circle with Luxembourg's (442) projected area × `t.k`, clamped to [1px, 10px] (cap reached ~12× effective zoom; the white halo thins below 3px radius). `luxAreaPx` is remeasured in `buildPaths` on every fit, so the projection scale is baked in and `t.k` alone gives the current screen size — a deep region fit (ミニ国家, ~20× effective) starts at the 10px cap while a world-wide fit (太平洋の島国) starts as a ~1px speck, matching what a real country that size would look like. Markers grow linearly with zoom like real land, and NEVER exceed Luxembourg at the same zoom (the owner's chosen honesty benchmark); if 442 ever leaves the data, a log2 fallback curve applies. The marker shows while `microMarkerVisible(id, t)` — `sqrt(projected polygon area)·t.k < MICRO_HIDE_PX` (area, not bbox, because multi-island spreads like Micronesia and the dateline-split Kiribati have huge bboxes; Vatican-class areas are ~0 so their marker never hides, which is correct). `featureAt` tests the same circles FIRST (same `microMarkerVisible`/`microR` selectors — the drawn circle IS the tappable circle, extending the pixel-exact-tap invariant), picking the nearest within `microR + MICRO_TAP_PAD`; without this priority the enclave states (Vatican/San Marino/Monaco…) would be swallowed by the surrounding country's polygon. `applyFocus` special-cases micro ids to a pseudo-bounds ±0.75° around the anchor (real bounds are useless: sub-pixel for Vatican, world-wide for split Kiribati); non-micro dateline crossers now focus via `mainPolygonIfSplit`. Browse labels for micro ids anchor at the marker (offset 13px below) with a synthetic 0.8° width so the name appears at a sane zoom. Markers are screen-space like labels, so blit gestures stretch them inside the snapshot and skip them in the exposed ring — same accepted artifact, cleared by the next `render()`.

**Click hit-testing** (`featureAt`, used by find mode and browse) has no DOM targets. It is pixel-exact: it tests the pointer against the **same cached Path2D** objects `render()` draws (via the same `activePaths()` LOD selector), under the *same* `dpr`+zoom transform, with `ctx.isPointInPath(activePaths().get(id), mx*dpr, my*dpr)` (backing-store px). A near-miss fallback strokes each path with a fat pen (`lineWidth = 16/t.k`, a constant ~8 CSS px) and uses `ctx.isPointInStroke`, picking the smallest-area country so tiny islands beat big neighbours. This deliberately avoids `projection.invert` + `d3.geoContains`, whose iterative inverse and great-circle edges disagree with the rendered straight-segment borders by a pixel or two (that mismatch was a real off-by-a-bit tap bug).

**Regions** come in two layers: the 7 coarse continents live in `countries.js` (`region` field), while finer sub-regions (東南アジア, 東ヨーロッパ, 北アフリカ, カリブ, 太平洋の島国, …) are defined in `app.js` as `SUBREGIONS` — keyed by sets of ISO numeric codes so `countries.js` stays untouched. Sub-regions marked `skipRev: true` (micro_europe ミニ国家（microstate） — a study set whose ids overlap the geographic sub-regions) are excluded from the `SUBREGION_OF` reverse map, so distractor rings always come from the geographic layer. (「ミニ国家」 is the established Japanese rendering of *microstate* — ja.wikipedia/コトバンク both use it — kept with the English gloss per the owner's preference.) `inRegion(f, r)` resolves either layer. The setup screen shows them as a **two-level accordion** (v28, progressive disclosure): `.region-top` holds 世界全体 + the continents, and every continent chip (`data-group`, no `data-region`) is **expand-only**: tapping it opens its `.region-sub` tray `#sub-<group>` (one open at a time; re-tap closes) whose first chip is the 〇〇全体 selector (アメリカ has no single region key, so its tray goes straight to 北アメリカ/中央アメリカ/カリブ/南アメリカ; オセアニア gained a tray in v31: オセアニア全体 + 太平洋の島国; 世界全体 is the only direct top-level selector). `chipGroup` owns the tap logic, `activateChip` re-derives open/active state when restoring saved settings, and `.chip.open` marks the expanded continent. Because the active chip can be hidden inside a *closed* tray (pick 北アフリカ, then open アジア's tray), the selection is always echoed twice: `#region-current` next to the 地域 label shows the selected region's name (`updateRegionCurrent`/`regionDisplayName`), and `.chip.holds` (via `syncRegionHolds`) tints the continent chip whose tray contains the active selection — never remove these without another always-visible selection indicator.

**Mastery & ranks (v38)**: per-country stats are `{ c, w, last, h }` — `c`/`w` are lifetime tallies (kept for the detail panel and `weightFor` question weighting), while `h` is the recent right/wrong history (1/0, newest last, capped at `HISTORY_MAX` 20). **The 成績マップ color comes from the last `MASTERY_WINDOW` (6) entries of `h`**, not lifetime rate — so a country flunked long ago recovers with ~5 straight corrects, and one slip after mastery keeps 5/6 green. `migrateStats` lazily backfills `h` on load for v1 records (proportional split, wrongs placed oldest so they fall out first). Rank tiers (`RANK_TIERS`, by green-bucket count): 0 ちず見習い / 10 まちある記 / 30 旅人 / 60 冒険家 / 100 地理博士 / 150 世界の達人 / 185 世界マスター / 200 全国家制覇.

**Browse/study mode** (`state.browsing`, mode `browse`) is a non-quiz view: `startBrowse` shows the region's map with country-name labels drawn on the canvas (`buildLabels` precomputes centroids/widths per fit; `render()` draws them unscaled in screen space, hiding labels for countries too small at the current zoom). Tapping a country (`showBrowseDetail`) reuses the explanation panel to show capital + blurb.

**First-launch tutorial** (`maybeStartTutorial`, v29): after the world loads, a first-time user gets a 3-step hands-on tutorial instead of the setup screen — welcome card (scrim overlay `#tutorial`, z-index 31) → れんしゅう① find practice (tap オーストラリア `036`; misses flash the tapped country red + name it, 2 misses make the target pulse so nobody stalls) → れんしゅう② name practice (日本 `392` highlighted+focused, 4 choices via `pickDistractors`, wrong picks just disable that button) → done card with confetti + mode summary (both practiced modes annotated) + a `.tut-reco` study-path tip (start with a small region like 東南アジア, widen gradually — the same advice echoed as a `field-hint` under the setup screen's region picker) →「次へ」→ install-recommendation card (`#tut-install-card`, same platform copy as the setup card's `.setup-install`; skipped entirely when already running standalone — the done card's button then reads はじめる！ and finishes directly) → `showSetup`. It borrows the quiz's `prompt-bar`/`choices` UI (restoring `choicesKicker` on exit) but **never touches `stats`/`mistakes`** and taps route through `onTutorialTap` (guard at the top of `onCanvasClick`). A floating `#tutorial-skip` chip (map steps) plus skip buttons on both cards all exit via `finishTutorial`. First-launch detection: `wq.tutorial.v1` flag; users with existing saved settings/stats are auto-flagged as veterans and never see it. Replayable anytime via the `#tutorial-replay` link at the bottom of the setup card.

**Offline:** `sw.js` caches the app shell + CDN libs on install (local assets must all succeed; CDN cached best-effort), then serves cache-first with a navigation fallback to `index.html`.

## Notes

- The setup card is ordered for low first-glance cost (v28): mode → region → 問題数 → the スタート button → a collapsed `<details id="more-settings">` holding 解説/効果音 whose summary line (`#more-values`, kept fresh by `updateSettingsSummary`) shows the current values without expanding; `onModeChange` hides the whole details block (and `#count-field` separately) for browse/progress. Keep new session-preference fields inside this details block, not above the start button.
- The setup card carries a `.setup-credit` block (below the lede, above the 称号 line) with the map-data attribution (world-atlas / Natural Earth — country coverage and borders follow the source data, 200 countries and territories; the Japanese names / capitals / blurbs are the app's own editing) plus the territorial-neutrality / accuracy disclaimer, a `.setup-install` box (below the version line) explaining how to add the PWA to the home screen on Android Chrome / iOS Safari / desktop Chrome+Edge, and a `.setup-legal` block at the very bottom of the card explaining that too-small-to-see countries are shown as circle markers and that Tuvalu is position-only (no polygon in the source data). The 18 former exclusions are all included since v31 via the marker system. Since v40 (beta release) the app carries a `.beta-badge` (β) on the topbar brand and the setup title, and a `.setup-feedback` box (between the tutorial-replay link and `.setup-install`) pointing feedback to the repo's GitHub Issues; `index.html`'s head has meta description + OGP tags whose og:url/og:image point at the GitHub Pages deployment.
- `countries.js` and `explanations.js` are marked "auto-generated" from `gen_countries.py` / `gen_icons.py`, but those generator scripts are **not in the repo** — edit the `.js` data files directly.
- Deploy target is GitHub Pages with `index.html` at repo root (see README).
- The map uses the 50m atlas as its detailed source, plus the 110m atlas (`WORLD_URL_LOW`) as the low-detail LOD set fetched in parallel; both must be precached by `sw.js` for full offline (under `MAP_POV="jpn"` they are LOCAL_ASSETS `./countries-*-jpn.json`; under `"default"` they are the world-atlas CDN URLs). 110m omits polygons for several small island nations — those fall back to their 50m Path2D in the low-detail set, so nothing disappears.
